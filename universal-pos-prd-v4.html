1. Executive Summary

1.1 Vision

The Universal Restaurant POS System is a zero-latency, hardware-native point-of-sale platform built for every restaurant type — from single-location quick service to 500-branch chains. Three non-negotiable guarantees:

  • Zero Latency: Orders created locally in under 50ms. The network is never on the critical path.
  • Offline First: Every terminal operates at 100% capacity without internet. Auth, permissions, and order-taking all work offline.
  • Hardware Native: Each client uses its own native rendering stack. No webviews. No Electron. No performance ceiling.

1.2 What v4.0 Adds

This version adds the complete Authentication and Permissions system — the most critical missing piece from previous versions. A POS that cannot control who does what is not production-ready.

  • Full staff auth: PIN, NFC card, Face ID / Touch ID per platform
  • Offline authentication: terminals verify identity without internet
  • JWT lifecycle: access tokens, refresh tokens, expiry, mid-shift handling
  • How to create and manage roles and permissions from the admin dashboard
  • Branch-scoped RBAC: a Branch A manager cannot see Branch B data
  • Custom roles: chains define their own role names and permission sets
  • Multi-role staff: a person who serves weekdays and manages on weekends
  • Manager override: approve a void without logging out current session
  • Temporary permission elevation with PIN and automatic expiry
  • Time-based access: staff only active during scheduled shift hours
  • Full audit log: every action stamped with who did it, when, and from which terminal

5. Authentication & Permissions

This is the complete specification for staff authentication, JWT token lifecycle, role-based permissions, branch scoping, custom role creation, manager overrides, and the audit system. Every terminal client and the backend must implement this spec precisely.

5.1 Staff Login Methods

POS terminals are shared devices. Staff do not use email/password. Login must be fast, work offline, and be practically impossible to accidentally log in as someone else. Three methods are supported:

Method

Security: PINs are never stored in plaintext. Hashed with bcrypt (cost 10) on the server. A PBKDF2 derivative is stored in the local SQLite token cache for offline PIN verification only.

5.2 Online Authentication Flow (PIN)

When the terminal has internet connectivity, the full auth flow runs against the NestJS Auth Service:

STEP 1 — Staff enters PIN on terminal login screen

STEP 2 — Terminal sends:

POST /api/v1/auth/login

Body: { staffId: "uuid", pin: "1234", terminalId: "uuid", branchId: "uuid" }

STEP 3 — Auth Service:

a. Look up staff record in Supabase

b. Verify bcrypt(pin, storedHash) = true

c. Check staff is active and shift is currently valid

d. Load staff roles and permissions for this branch

e. Issue JWT pair:

accessToken: { staffId, branchId, roles[], permissions[], exp: now+15min }

refreshToken: { staffId, branchId, exp: now+8hr (one shift) }

STEP 4 — Response to terminal:

{ accessToken, refreshToken, staffProfile, roles[], permissions[] }

STEP 5 — Terminal stores in encrypted SQLite:

{ accessToken, refreshToken, pinHash (PBKDF2), roles, permissions, expiresAt }

STEP 6 — Terminal renders home screen filtered by the staff member's permissions

5.3 Online Authentication Flow (NFC Card)

STEP 1 — Staff taps NFC card on reader

STEP 2 — Terminal reads card UID (unique hardware identifier)

STEP 3 — Terminal sends:

POST /api/v1/auth/login/nfc

Body: { cardUid: "hex-string", terminalId: "uuid", branchId: "uuid" }

STEP 4 — Auth Service:

a. Look up staff record by cardUid in Supabase

b. Verify card is active and not revoked

c. Check shift validity

d. Issue JWT pair (same as PIN flow above)

STEP 5–6 — Same as PIN flow

5.4 Online Authentication Flow (Face ID / Touch ID — iOS)

STEP 1 — Staff taps their avatar on the iPad login screen

STEP 2 — iOS LocalAuthentication framework triggers Face ID / Touch ID prompt

STEP 3 — On biometric success, retrieve stored refreshToken for this staff member

from the iOS Keychain (encrypted hardware-backed storage)

STEP 4 — Terminal sends:

POST /api/v1/auth/refresh

Body: { refreshToken: "stored-token" }

STEP 5 — Auth Service issues new accessToken (15min)

STEP 6 — Terminal is now logged in — no PIN required

Note: The refreshToken never leaves the iOS Keychain in plaintext. It is retrieved by the OS only after biometric success. This means Face ID is essentially a gate to the refresh token, not to the server.

5.5 Offline Authentication Flow

When the terminal has no internet, it cannot call the Auth Service. The offline auth flow uses the locally cached token and a PBKDF2 hash of the PIN:

STEP 1 — Terminal detects OFFLINE_MODE (3s WebSocket timeout)

Status bar shows amber offline indicator

STEP 2 — Staff enters PIN on login screen

STEP 3 — Terminal looks up staff in local SQLite offline_auth_cache:

SELECT * FROM offline_auth_cache WHERE staffId = ?

STEP 4 — Verify PIN locally:

a. Compute PBKDF2(enteredPin, storedSalt, 100000 iterations, SHA-256)

b. Compare to stored PBKDF2 hash in SQLite

c. If match: PASS. If no match: increment failCount.

STEP 5 — If failCount >= 5:

a. Lock terminal for this staff member for 60 seconds

b. Log failed attempt in offline_audit_log

STEP 6 — If PIN verified:

a. Decode cached JWT (no network call needed — just decode the payload)

b. Check cached roles and permissions

c. If cachedToken.exp < now AND > (now - 8hr grace):

Allow login with cached permissions (grace period)

d. If cachedToken.exp < (now - 8hr):

Reject — token too stale, manager PIN required to unlock

STEP 7 — All actions taken offline are tagged: { offlineAuth: true }

These are included in the offline_log and synced on reconnection

STEP 8 — On reconnect: full online auth flow runs silently in background

Fresh tokens replace cached tokens

Security: The 8-hour grace period matches a single shift length. A token older than 8 hours offline means the terminal has been disconnected for over a shift — that is abnormal and requires manager intervention.

2. Terminal clears accessToken from memory

4. Server adds refreshToken to Redis blacklist (TTL = remaining expiry)

5. Terminal returns to login screen

3. Any open orders are flagged for handover

1. Manager clicks "Force Logout" on staff member in admin

2. Server blacklists all active tokens for that staffId

3. Terminal receives logout push via WebSocket sync gateway

4. Terminal clears session immediately

5.7 Role Design

The system uses a three-layer permission model: System Roles (built-in, cannot be deleted), Branch Roles (manager-created per branch), and Custom Chain Roles (owner-created, available across all branches).

Layer 2: Custom Roles (Owner/Admin creates these)

Owners and admins can create custom roles that extend or restrict the built-in roles. Custom roles are named by the chain (e.g. "Head Waiter", "Shift Lead", "Bar Manager") and are built by selecting individual permissions from the permission catalogue.

Layer 3: Multi-Role Staff

A staff member can hold multiple roles, either globally or per branch. The effective permission set is the union of all assigned roles. Example:

Ali: roles = [SERVER (Branch A), BRANCH_MANAGER (Branch A — weekends only)]

On a weekday: Ali has SERVER permissions

On a weekend: Ali has BRANCH_MANAGER permissions (union wins)

Permission resolution: effectivePermissions = UNION(role1.perms, role2.perms, ...)

5.9 Permission Evaluation Logic

The RBAC Guard on every NestJS endpoint follows this evaluation sequence:

RBAC GUARD — runs on every API request

──────────────────────────────────────────────────────────────

1. Extract JWT from Authorization header

2. Verify JWT signature (using public key)

3. Check JWT is not in Redis blacklist (logged out / force revoked)

4. Extract { staffId, branchId, roles[], permissions[] } from payload

5. Extract required permission from route decorator:

@RequirePermission('orders:void')

6. Check branch scope:

Is the requested resource in the same branchId as the token?

If branchId mismatch AND role is not OWNER/ADMIN/AUDITOR → 403

7. Check permission:

Is 'orders:void' in token.permissions[]? → ALLOW

Is it not? → 403 Forbidden

8. Log action to audit service:

{ staffId, action, resource, branchId, terminalId, timestamp, result }

──────────────────────────────────────────────────────────────

All 403s are logged. Three 403s in 60s from same terminal triggers alert.

5.10 Branch Scoping

Branch scoping is the rule that staff can only see and act on data within their assigned branch. It is enforced at three independent layers:

  • Layer 1 — JWT Payload: The accessToken embeds branchId. A token issued for Branch A cannot be used to access Branch B resources.
  • Layer 2 — RBAC Guard: Every API endpoint checks that the requested resource's branchId matches the token's branchId unless the role is OWNER, ADMIN, or AUDITOR.
  • Layer 3 — Supabase RLS: Row Level Security policies on the database reject queries where the session's branch_id does not match the row's branch_id. This is the last line of defence even if the API guard is somehow bypassed.

-- Supabase RLS policy example (orders table)

CREATE POLICY "Branch isolation on orders"

ON orders

FOR ALL

USING (

branch_id = current_setting('app.current_branch_id')::uuid

OR

current_setting('app.staff_role') IN ('OWNER', 'ADMIN', 'AUDITOR')

);

5.11 Manager Override Flow

When a server needs to perform an action above their permission level (e.g. apply a discount, void an item), they do not need to log out. The manager approves inline without losing their own session:

MANAGER OVERRIDE FLOW

──────────────────────────────────────────────────────────────

1. Server taps "Apply Discount" (requires orders:discount permission)

2. Terminal checks: server does NOT have orders:discount

3. Terminal shows OVERRIDE SCREEN:

"This action requires manager approval.

Manager PIN or NFC card required."

4. Manager enters their PIN or taps NFC card on the same terminal

5. Terminal sends:

POST /api/v1/auth/override

Body: { managerPin: "xxxx", action: "orders:discount",

orderId: "uuid", staffId: "current-server-uuid" }

6. Auth Service:

a. Verify manager PIN

b. Check manager has orders:discount

c. Issue short-lived OVERRIDE TOKEN: { exp: now+60s, action: "orders:discount" }

7. Terminal receives overrideToken

8. Terminal executes the action using overrideToken (single use)

9. Audit log records:

{ action: "orders:discount", performedBy: serverStaffId,

approvedBy: managerStaffId, overrideToken: used }

10. Server's session unchanged — they remain logged in as SERVER

──────────────────────────────────────────────────────────────

Override tokens expire in 60 seconds and are single-use only.

5.12 Temporary Permission Elevation

For recurring scenarios where a staff member regularly needs a specific elevated permission, a manager can grant temporary elevation with an automatic expiry — without creating a new role:

EXAMPLE: Promote Ali to Shift Lead for tonight only

In admin dashboard:

Staff → Ali → Temporary Elevation

Permission: orders:discount, orders:void_item

Valid from: 17:00 today

Expires: 23:59 today

Approved by: Manager (requires manager auth to save)

Server-side effect:

A temporary_permissions record is created in Supabase:

{ staffId, permissions[], branchId, validFrom, validUntil, grantedBy }

Token refresh behaviour:

On next token refresh, Auth Service includes temp permissions in JWT

Permissions are automatically excluded after validUntil

Audit trail:

All actions taken under temporary elevation are flagged in audit log

5.13 How to Create and Manage Roles

Roles are created and managed in the Admin Dashboard under Staff > Roles. Only OWNER and ADMIN can create roles. Roles can be scoped to a single branch or made available chain-wide.

Creating a New Role — Step by Step

  1. In the Admin Dashboard, navigate to Staff > Roles > Create New Role
  2. Enter a role name (e.g. "Head Waiter", "Shift Lead", "Bar Manager")
  3. Select scope: This Branch Only OR All Branches
  4. Select base template: Start from scratch OR clone an existing role
  5. Toggle individual permissions from the Permission Catalogue (Section 5.8)
  6. Set optional restrictions: max discount %, max void amount, time-of-day restrictions
  7. Click Save — role is immediately available to assign to staff

7. Data Synchronization Architecture

7.1 Three Core Guarantees

  • Availability over Consistency: A branch takes orders regardless of internet. Auth works offline.
  • Eventual Consistency: All branches converge to identical state on reconnection.
  • No Data Loss: Every offline operation is persisted to an immutable log before returning success to the UI.

7.3 Offline Survival Flow

NORMAL OPERATION (Connected)

Terminal ──WebSocket──► Sync Gateway ──► Supabase PostgreSQL

◄── Real-time state updates ◄──────────────────────

OUTAGE DETECTED (3-second WebSocket timeout)

1. SyncService sets OFFLINE_MODE = true

2. Amber dot appears in terminal status bar

3. All mutations → append to SQLite offline_log with vector clock

4. Auth continues using cached tokens (up to 8hr grace period)

RECONNECTION

5. WebSocket reconnects, full online auth runs silently

6. CRDT merge resolves any conflicts

7. Audit logs for offline actions synced to server

8. Fresh tokens replace cached tokens

7.4 Reference Vector Clock

static merge(a: VectorClock, b: VectorClock): VectorClock {

const merged: VectorClock = {};

const allKeys = new Set([...Object.keys(a), ...Object.keys(b)]);

for (const key of allKeys) {

merged[key] = Math.max(a[key] ?? 0, b[key] ?? 0);

}

return merged;

}
