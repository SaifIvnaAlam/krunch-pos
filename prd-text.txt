1. Executive Summary
1.1 Vision
The Universal Restaurant POS System is a zero-latency, hardware-native point-of-sale platform built for every restaurant type ΓÇö from single-location quick service to 500-branch chains. Three non-negotiable guarantees:

Zero Latency: Orders created locally in under 50ms. The network is never on the critical path.
Offline First: Every terminal operates at 100% capacity without internet. Auth, permissions, and order-taking all work offline.
Hardware Native: Each client uses its own native rendering stack. No webviews. No Electron. No performance ceiling.

1.2 What v4.0 Adds
This version adds the complete Authentication and Permissions system ΓÇö the most critical missing piece from previous versions. A POS that cannot control who does what is not production-ready.

5.10 Branch Scoping
Branch scoping is the rule that staff can only see and act on data within their assigned branch. It is enforced at three independent layers:

Layer 1 ΓÇö JWT Payload: The accessToken embeds branchId. A token issued for Branch A cannot be used to access Branch B resources.
Layer 2 ΓÇö RBAC Guard: Every API endpoint checks that the requested resource's branchId matches the token's branchId unless the role is OWNER, ADMIN, or AUDITOR.
Layer 3 ΓÇö Supabase RLS: Row Level Security policies on the database reject queries where the session's branch_id does not match the row's branch_id. This is the last line of defence even if the API guard is somehow bypassed.

5.13 How to Create and Manage Roles
Roles are created and managed in the Admin Dashboard under Staff > Roles. Only OWNER and ADMIN can create roles. Roles can be scoped to a single branch or made available chain-wide.

Creating a New Role ΓÇö Step by Step
In the Admin Dashboard, navigate to Staff > Roles > Create New Role
Enter a role name (e.g. "Head Waiter", "Shift Lead", "Bar Manager")
Select scope: This Branch Only OR All Branches
Select base template: Start from scratch OR clone an existing role
Toggle individual permissions from the Permission Catalogue (Section 5.8)
Set optional restrictions: max discount %, max void amount, time-of-day restrictions
Click Save ΓÇö role is immediately available to assign to staff

5.16 Audit Log
Every action on every terminal is recorded in the immutable AuditLog table. The audit log cannot be edited or deleted ΓÇö only read. It is the source of truth for any dispute, investigation, or compliance check.


Compliance: The audit log satisfies PCI-DSS requirement 10 (track and monitor all access to network resources and cardholder data) and supports internal HR investigations.
